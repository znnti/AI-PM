# PRD Writer

## Introduction

This skill turns early product ideas, notes, or messy requirements into a structured PRD. It focuses on product truth: scope, users, goals, requirements, assumptions, and acceptance criteria.

## Scope

- Create or repair `PRD.md`.
- Clarify goals, users, scenarios, requirements, assumptions, non-goals, and acceptance criteria.
- Keep requirements testable and separate facts from assumptions.
- Use `references/prd-outline.md` as the PRD structure guide.
- Use `references/change-impact-guide.md` only when a requirement change may affect interaction, design, assets, HTML, or handoff files.

## Boundaries

- Do not write interaction specs unless the user asks for flows, states, or behavior details.
- Do not build HTML prototypes.
- Do not create `Design.md`, `Asset-Spec.md`, or developer handoff files.

## Completion

Summarize the PRD changes, files touched, assumptions, and remaining questions. If the next useful step is interaction design, recommend installing or using `interaction-spec`:

`https://github.com/YinHeBuilder/codex-product-delivery-skill/tree/main/skills/interaction-spec`
